为什么析构函数不执行？让我们顺着你刚才的推理，揭开这个**“人质困境”**：

### 困境重演：无处安放的智能指针

在 C++ 中，`std::shared_ptr` 的析构原则是：**引用计数（ref_count）必须等于 0，才会触发对象的析构（`~Fiber()`）**。

我们来盘点一下，测试结束那一刻，你的 `Fiber` 对象的引用计数是多少：

1.  **主线程的持有者：**
    在 `TEST(FiberTest, SwapSpeed)` 中，你写了 `sylar::Fiber::ptr fiber(new ...);`。测试结束时，这个局部变量 `fiber` 确实超出了作用域，它的析构函数被调用，**引用计数 -1**。
    
2.  **子协程栈上的“幽灵持有者”：**
    这也是刚才让你闭上眼睛想象的地方。子协程永远停在了最后一次 `yield()` 内部。
    请看你之前自己写的 `src/fiber.cpp` 里的 `mainFunc()`：
    ```cpp
    void sylar::Fiber::mainFunc() {
        sylar::Fiber::ptr cur = GetThis(); // <--- 【关键点】这里产生了一个 shared_ptr
        cur->m_cb();                       // <--- 子协程被冻结在这里面的 yield() 里了
        
        // ... 下面的代码因为 yield 没有切回来，永远不会被执行
        cur.reset(); 
        raw_ptr->yield();
    }
    ```
    因为子协程被“冻结”了，它的执行流永远没有走出 `mainFunc` 这个函数。
    按照 C++ 的规则，**只有当函数执行完毕退出作用域时，局部变量 `cur` 才会被销毁**。
    所以，这个 `cur` 就像一个“幽灵”一样，永远驻留在那个 128KB 的协程栈里，死死地把持着 `Fiber` 的生命，导致 **引用计数始终 >= 1**。

**1. 框架设计层面（你的 `Fiber` 类设计有问题吗？）**

是的！在标准的 C++ 协程范式中，**强制要求所有的协程必须运行到结束状态（TERM）**，除非你提供了类似强杀（Cancel）的机制。
如果一个协程还在半路挂起（HOLD 状态），你就把外面主线程的智能指针丢弃了，这就是典型的**“遗弃孤儿协程”**，在任何基于 `shared_ptr` 的协程库中都会导致内存泄漏。

### 1. 为什么不能直接暴力“销毁”挂起的协程？

你可能会想：既然我要退出程序了，我直接在 `Fiber` 里面加个暴力销毁的方法，把那 128KB 栈内存 `delete[]` 掉不就行了吗？

**在 C++ 中，这是灾难。**
如果协程挂起在半路，它的栈上可能不仅有 `shared_ptr`，还可能有 `std::mutex::scoped_lock`、打开的数据库连接对象、文件句柄等。
如果你直接把栈内存回收了，这些对象的**析构函数将永远不会被调用**。这不仅会导致内存泄漏，还会导致死锁、连接池耗尽等极其严重的系统级故障。（这被称为 C++ 的 Stack Unwinding 栈展开问题）。

因此，C++ 协程界有一个铁律：**协程必须自己走完自己的一生（正常结束，或者抛出异常结束），框架决不能从外部直接“拔管子”强杀。**

---

### 2. 破局之道：引入状态机 (State)

正如你所说，必须给协程加上状态。标准的协程状态通常有这几种：
*   `INIT`：刚创建，还没运行。
*   `EXEC`：正在运行。
*   `HOLD`：挂起等待中（调用了 yield）。
*   `TERM`：正常执行完毕。
*   `EXCEPT`：因异常崩溃结束。

有了状态，我们就可以在 `fiber.cpp` 中做文章了。比如在 `mainFunc()` 的最后：
```cpp
void sylar::Fiber::mainFunc() {
    sylar::Fiber::ptr cur = GetThis();
    cur->m_state = EXEC;
    try {
        cur->m_cb();
        cur->m_state = TERM; // <--- 关键！执行完标记为死亡
    } catch (...) {
        cur->m_state = EXCEPT;
    }
    cur->m_cb = nullptr; 
    sylar::Fiber* raw_ptr = cur.get();
    cur.reset();
    raw_ptr->yield(); // 最后一次切回去
}
```

### 3. 终极Boss：调度器 (Scheduler) 的引入

当你的系统里有几万个协程时，你是不会在业务代码里自己去 `new Fiber()` 和 `fiber->resume()` 的。

所有协程的生命周期，全部交给 **调度器（Scheduler）** 来接管。调度器内部维护了一个**任务队列**。

调度器的工作原理就是一个死循环（Main Loop）：
1. 从队列里拿出一个就绪状态的协程。
2. 调度器调用 `fiber->resume()`，交出 CPU 控制权。
3. 协程在里面可能执行了很久，遇到了 IO 阻塞，调用了 `yield()`（此时状态变为 `HOLD`）。CPU 切回调度器。
4. **调度器检查刚切回来的这个协程的状态：**
    * 如果是 `HOLD`，调度器不管它，继续去队列里拿下一个协程。
    * 如果是 `TERM` 或 `EXCEPT`，调度器就知道这个协程死了。调度器会从内部结构中抹除对它的引用，智能指针计数归零，**自动触发析构函数，回收内存！**

### 4. 那么，程序退出时那些还在 HOLD 的协程怎么办？

这才是你最关心的问题：程序都要退出了，队列里还有几百个 `HOLD` 的僵尸协程怎么办？

这其实是**业务层的职责，而不是底层机制的缺陷**。
标准的处理方式是：**协作式取消（Cooperative Cancellation）**。

业务代码需要在内部检查退出标志位：
```cpp
void fiber_speed_func() {
    for(int i = 0; i < SWAP_N; i++) {
        if (g_is_shutting_down) {
            break; // 收到退出信号，立刻打断循环，走向函数结尾
        }
        sylar::Fiber::GetThis()->yield();
    }
    // 顺利走到这里，进入 TERM 状态
}
```

在系统关闭时，调度器会设置 `g_is_shutting_down = true`，然后**最后一次唤醒所有 HOLD 状态的协程**（把它们全都 resume 一遍）。
这些协程醒来后，看到标志位，就会立马结束执行流，走向 `TERM` 状态。最终完美回收，滴水不漏。

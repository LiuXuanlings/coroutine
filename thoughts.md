# C++ 协程调度器 (Scheduler) 设计备忘录

### 一、 基础设施与生命周期管理

1.  **文件结构与核心成员**
    *   创建 `Scheduler.h` 和 `Scheduler.cpp`。
    *   **线程池管理：** 在 `Scheduler` 类中引入 `std::vector` 作为私有容器，存储底层线程（封装好的 `Thread` 类的智能指针）。
    *   **容量设定：** 现阶段采用硬编码，将线程池大小固定为 32。
    *   **初始化：** 在 `Scheduler` 的构造函数中，通过 `for` 循环遍历 32 次，实例化并启动这些线程。

2.  **全局单例模式 (Singleton) 的无锁化与安全**
    *   使用 `std::atomic<Scheduler*>` 类型的静态变量 `t_scheduler` 保存实例。
    *   **双检锁机制 (Double-Checked Locking)：**
        *   第一层检查：使用 `load` 配合 `memory_order_acquire` 内存序读取到局部变量，若不为空直接返回。
        *   若为空，则进入互斥锁保护的代码块。
        *   第二层检查：在锁内再次判断是否为空（防止并发初始化）。若确认为空，则执行 `new Scheduler()`，并完成 32 个线程的构造。

### 二、 核心运转机制：生产者-消费者模型

1.  **消费者逻辑：线程主循环 (`run` 函数)**
    *   该函数是 32 个线程的共同启动入口。
    *   **资源保护：** 引入 `std::queue` 作为任务队列，并配以私有变量 `std::mutex` 和 `std::condition_variable (cv)` 保证多线程并发访问的安全。
    *   **循环流转：**
        *   进入循环即加锁 `mutex`。
        *   检查任务队列是否为空。
        *   若**为空**：调用 `cv` 等待，自动释放锁并阻塞挂起当前线程。
        *   若**不为空**：从队头 `pop` 出一个任务（Fiber 实例），随后调度并执行该任务（注：需在释放锁之后执行，避免阻塞队列）。

2.  **生产者逻辑：任务分发 (`schedule` 函数)**
    *   **接口设计优化：** 摒弃仅接收单一 Fiber 指针的设计。改为接收**函数指针 + 变长参数**。在内部使用 `std::bind` 将其封装为 `Fiber` 对象，然后再进行入队操作。
    *   **入队流转：**
        *   对队列加锁。
        *   记录入队前“队列是否为空”的状态到一个局部标志位。
        *   将封装好的 Fiber 任务 `push` 进队列。
        *   离开作用域，释放锁。
    *   **唤醒策略：** 检查刚刚记录的标志位。如果入队前队列为空，说明此时工作线程大概率都在 `cv` 上睡眠等待。此时只需调用 `notify_one()` 唤醒其中一个线程来处理新任务即可，无需惊动所有线程。

### 三、 优雅退出与清理机制 (`stop` 函数)

1.  **退出状态机**
    *   在类中新增私有布尔变量 `is_stopping`，默认值为 `false`。
    *   调用 `stop()` 函数时，首要操作是将 `is_stopping` 置为 `true`。
    *   **联动 `run` 函数：** 修改 `run` 函数的循环条件。在每次加锁检查时，若满足 `is_stopping == true` **且** `任务队列为空`，则触发 `break`，彻底退出 `run` 循环。

2.  **资源回收与待解决疑问**
    *   **设计留白（原思路中的关键疑问）：** `stop` 函数将状态置为 `true` 后，如何准确得知所有 32 个线程都已经完全退出了 `run` 循环？（注：后续 Claude 提示此处需使用 `thread->join()` 来阻塞等待）。
    *   **清理动作：** 确认所有线程退出工作循环后，遍历线程池 vector，调用内部 `shared_ptr` 的 `reset()` 方法释放线程对象，完成调度器的销毁。

### 四、 正确性验证 (Test Suite)

1.  **测试用例设计**
    *   编写两个模拟业务函数：
        *   Task A：执行简单的标准输出打印。
        *   Task B：执行 `sleep` 模拟耗时 I/O，随后打印。
2.  **验证流程**
    *   调用 `GetInstance()` 获取单例。
    *   在循环中，将 Task A 和 Task B 各自 `schedule` 提交 100 次。
    *   投递完成后立即调用 `stop()`。
    *   预期结果：程序能够稳妥阻塞至 200 个任务全部执行完毕，并安全退出。
